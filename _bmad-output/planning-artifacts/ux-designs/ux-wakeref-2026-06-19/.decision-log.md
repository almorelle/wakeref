## Context inherited (from conversation + project-context.md)

- **Feature**: a judge-training module for WakeRef. Given an existing full-run video, the learner enters the run using the existing Compo saisie UI, then their input is compared against a single fixed reference solution (one video = one solution). Output is a correction shown as an element-by-element diff.
- **Core intent (locked by user)**: compare the *saisie* (the entered run), NOT the calculator score. A run video has exactly one fixed correct saisie; the learner's input must match it.
- **Reuse**: the Compo saisie component (`src/pages/Compo.jsx`). Entry shape = list of `entries` (`slug`, `category_slug`, `side` left/right, `approach[]` hs/ts/regular/fakie, rotation, `inverted`, `_seq` order) + `jibPasses` (expand to pseudo-entries). Add modes per grid: jib / kicker / air_trick / flat. Grid selected via `gridKey` (wakeboard, wakeskate, seated_mp1, seated_mp5).
- **Existing peer**: a Quiz module already exists.
- **Stack**: React 19 + Vite PWA, CSS Modules + global tokens in `src/index.css` (`[data-theme]` theming), Supabase (RLS only boundary). Single admin auth; no public sign-up. i18n fr/en. Mobile-first PWA.

## Decisions taken

### D1 — Diff semantics (from conversation, pre-design)
- Comparison is per-element, sequence aligned. Error types to surface: trick **missing**, trick **extra**, right trick / wrong **approach**, right trick / wrong **order**. (rotation/variant possible if stored.)
- DEFAULT proposed (not yet user-confirmed): alignment by position in the run; order matters for a judge.

### D2 — Reference solution storage (leaning, to confirm at architecture stage, not UX)
- Leaning toward a dedicated table (e.g. `judge_references`) rather than a flag on `compositions`: different lifecycle (admin-authored), different RLS (must NOT leak the solution to the learner before submit), linked to a video. Shares the Compo snapshot format.

### D3 — Form-factor: DESKTOP-FIRST (user decision)
- The judge analyses a run on a big screen; video + saisie side-by-side. Mobile is the degraded case, not the priority. (Notable departure from WakeRef's mobile-first default — flag for DESIGN.md.)

### D4 — IA placement: NEW TOP-LEVEL NAV ITEM (user decision)
- Sits alongside Quiz as a first-class feature (e.g. "Entraînement juge" / "Judge training"). Not a sub-mode of Compo, not under Quiz.

### D5 — Layout: VIDEO + SAISIE SIMULTANEOUS, NO SCORE CALCULATOR (user decision)
- Video stays visible/scrubbable while the judge enters each element.
- CRITICAL SCOPE: the Compo *score calculator* (the /20 grid panel) is NOT shown and NOT computed in this mode. Reuse only the saisie sub-component. The goal is purely: did the judge capture all the right tricks in the right order.
- Implication for reuse: the saisie UI must be extractable from Compo independent of GRIDS scoring. (Architecture note.)

### D6 — Post-submit: REVEAL THE SOLUTION (user decision)
- On validate → show the diff AND reveal the full reference solution alongside the judge's saisie. (Retry, progress tracking, trick-card links were NOT selected.)
- No per-user progress storage (consistent with no public auth). Training is stateless / anonymous.

### D1 confirmed — order matters
- User: "tous les bons tricks dans le bon ordre" → sequence-aligned diff, order is part of correctness.

### D7 — Run selection: BY DISCIPLINE + BY DIFFICULTY (user decision)
- Filter by discipline (wakeboard / wakeskate / seated), and ideally by difficulty bucket:
  - **easy** = U11 + MP1 à MP3
  - **medium** = U14 + MP3 à MP5 + O40
  - **hard** = U18 + open + O30
- Difficulty buckets map to competition categories. Implication: each reference run is tagged with its competition category, rolled up to easy/medium/hard. (Data note: this category/difficulty tag likely a new field on the reference record — not currently modeled.)
- Mechanism (catalog vs random) not yet locked; a filterable catalog (discipline + difficulty) is the through-line. Random "next run" optional, deferable.

### D8 — Correctness criterion: EXACT TRICK + EXACT ORDER (user decision, option 1)
- An element is "correct" only if it matches on all defining attributes (trick/slug, approach hs/ts/regular/fakie, side left/right, rotation) AND its position in the run.
- EXCEPTION: the free-text "autre" field in the saisie is NOT counted as a fault for now (excluded from the diff/error logic).

### D9 — Journey (user-narrated)
1. Judge opens the Judge Training module.
2. Chooses a **difficulty** + a **discipline**.
3. A run video plays; she enters tricks. Realistic note: she may pause, or even note on paper and enter everything after the run ends. Pausing/scrubbing is OK (not faithful to live judging, but accepted). Normal video player — she can re-watch a passage.
4. She clicks **"Évaluer mon run"**.
5. The correction appears **beside her run in diff mode** (reference solution revealed alongside her saisie).
6. The **video stays available** so she can review the moments where she erred.
7. She can **"Recommencer"** → returns to the difficulty + discipline choice (NOT an in-place re-edit; it's a fresh run selection).

Climax beat: step 4→5, the reveal — at a glance she must see what she missed, what she added in error, and where she misread an attribute (approach/side/rotation) or swapped order.
